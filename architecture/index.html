Architecture

Digital Twins with Asset Administration Shells

The Asset Administration Shell (AAS) is the major asset for creating digital twins. AAS sub models contain specific data models, state information, and provide access to asset specific services. Without them, the digital twin would be of little use. Eclipse BaSyx supports three types of AAS sub models. One AAS may consist of any number of sub models, and may realize all kinds of sub models:

  • Static sub models provide access to data that only changes seldomly. Documents, electronic certificates, data sheets, and name plates are examples for static sub model data. Static sub model data may be serialized and sent to suppliers or customers.
  • Dynamic sub models provide a unified access to asset data. This may be raw data, i.e. the state of a device, or higher-level data, such as the location of a product in the process. Dynamic sub models also support behavior (using program code) that collects, pre-processes, and analyzes data. They can also implement interfaces to other systems and system components. Dynamic sub models are realized with the Eclipse BaSyx Software Development Kits (SDKs.)
  • Control component sub models enable unified access to control components and thus to asset services and capabilities. Control components support controlling of a machine with a unified interface, and to query its state. Control components therefore are one foundation for the service-oriented approach of Eclipse BaSyx that enables efficient lot-size one production.

Eclipse BaSyx Architecture

Eclipse BaSyx realizes a p2p architecture that connects all relevant Industry 4.0 assets to a central virtual communication medium that enables end-to-end connectivity.

  • Enterprise level Applications on the enterprise level use digital twins to control, optimize and monitor the digital factory, and to negtiate with external stakeholders.

  • Middleware level The middleware consists of asset administration shells and sub models that together with dynamic sub model realize digital twins of e.g. processes, resources and products. Group control components control the manufacturing process by orcestrating services provided by shopfloor resources. The registry enables localization of AAS and sub models.

  • Digital process level The digital process level consists of dynamic sub models that reflect the current status of e.g. processes, products, and resources, as well as control components that provide a service interface for controlling resources and processes.

  • Process level Represents real or simulated assets. The term „Assets“ represent any kind of asset, which may be physical, non-physical, or existing systems such as ERP and MES systems, PLM systems, and databases. Assets interact with dynamic sub models and control components through OPC-UA, MQTT or field bus protocols.

AAS – Higher-level Interfaces

Asset Administration Shells provide a unified access to all assets of a manufacturing process. An AAS provides access to raw device data, e.g. by delegating requests to device OPC UA servers, and by providing a facade for e.g. IT systems to enable end-to-end communication.

While individual AAS sub models may provide raw device data access, the digital twin that is realized by an AAS puts focus on higher-level abstractions. For example, while the state (open/closed) of a light barrier is a raw data value, the AAS will aggregate this information to represent e.g. the product instance that is currently on a specific position in the factory.

AAS and AAS sub models may therefore create hierarchies to implement more abstract views and services that improve understandability and controllability of complex manufacturing environments, and that therefore lower maintenance and development cost.

Data Storage and Data Bridge

The Eclipse BaSyx data bridge connects real and simulated assets with dynamic sub models. It supports a variety of IoT, IIoT protocols, such as OPC UA, MQTT, http/REST, as well as field bus protocols with assets.

AAS sub models provide data through sub model properties and enable access to services. AAS sub models also optionally may define active functions, and therefore have the ability to preprocess raw data, and actively interact with other AAS and sub models. Sub models therefore can actively create higher-level views on processes, track products, and implement harmonized interfaces to assets and backend systems.

Using the MQTT protocol, sub models can notify subscribers about relevant changes in property values and other relavant events to avoid the need to constantly poll values.

Service-oriented Manufacturing Architecture

Eclipse BaSyx implements service-oriented manufacturing architectures. The service-ortiented architecture keeps the implementation of production steps separate from product recipes that deswcribe the necessary process for producing a product or a work piece.

Implementation of a service-oriented architecture (SoA) is based on the control component interface, and (hierarchically structured) group control components that orchestrate services. Both control components and group control components define a harmonized interface to access devices, operation modes, services, and to query the device status.

Orchestrators use product recipes that are defined in product AAS sub models. Recipes define required process steps that are necessary for creating a product. Sub models of the device AAS define capabilities, qualities offered production steps and associated cost. Optionally additional information, such as the required CO2 footprint may be added as well.

In a SoA, orcestrators evaluate product recipes, calculate schedules and production plans, and control the manufacturing steps for a product. To enable scalability, usually multiple orchestrators operate in parallel and in hierarchies. They use the unified control component interface to coordinate manufacturing steps for a product, thus enabling an efficient lot-size one production.

We provide the Camunda Engine as predefined orchestrator, and provide an interface to integrate other engines as orchestrators into BaSyx.

Eclipse BaSyx - Main Components

* Asset Administration Shell (AAS) Server component * Provides AAS to other components * Serialize/Load AAS from/to AASX, JSON, XML

* AAS Server component (for sub models) * Enables distributed deployment of sub-models * Support sub models with active code (e.g. preprocessing)

* AAS Registry component * Enables lookup and retrieval of AAS descriptors * WiP: Cascading of AAS registries

* Storage components * Persist data. Server/Registry components do not maintain a state.

* Ready-to-use applications for live data analysis * Eclipse Grafana integration (Dashboard application) * Streamsheets integration (Real-time Excel)

* Live event handling application * NodeRED integration (automated event processing)

* Device connectivity

  • Support for OPC-UA, MQTT, HTTP
  • WiP: PLC4X integration for fieldbus protocols

* True digital twins * Control components and group components * Enable controlling of processes and process steps
